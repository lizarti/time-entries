# API layer implementation

Build the full HTTP layer for the time-tracker domain: Resources, FormRequest with custom
rules, Actions, Controllers and Routes, backed by feature tests for every endpoint.

## Goal

All six endpoints will be reachable, correctly validated, and return consistent JSON shapes.
The BulkInsert action will enforce both DB-level and intra-batch business rules inside a
single transaction, rolling back everything on any failure.

## Target Structure

```
app/
  Actions/
    TimeEntry/
      BulkInsertTimeEntriesAction.php
  Http/
    Controllers/
      CompanyController.php
      EmployeeController.php
      ProjectController.php
      TaskController.php
      TimeEntryController.php
    Requests/
      BulkInsertTimeEntriesRequest.php
    Resources/
      CompanyResource.php
      EmployeeResource.php
      ProjectResource.php
      TaskResource.php
      TimeEntryResource.php
  Rules/
    BelongsToCompany.php
    EmployeeAssignedToProject.php

routes/
  api.php
```

## Steps

### Step 1 — API Resources

Create the five API Resources that serialize model data for every endpoint response.

- [x] `CompanyResource` — `id`, `name`
- [x] `EmployeeResource` — `id`, `name`
- [x] `ProjectResource` — `id`, `company_id`, `name`
- [x] `TaskResource` — `id`, `company_id`, `name`
- [x] `TimeEntryResource` — `id`, `company_id`, `employee_id`, `project_id`, `task_id`, `date` (Y-m-d string), `hours`

---

### Step 2 — FormRequest and custom rules

Create the two reusable Rule classes and the BulkInsertTimeEntriesRequest that applies them.

- [x] `BelongsToCompany` rule
  - Receives the model class and the `company_id` from the same entry
  - Fails if the given record does not belong to that company
- [x] `EmployeeAssignedToProject` rule
  - Receives `employee_id` and `project_id` from the same entry
  - Fails if no row exists in `employee_project` for that pair
- [x] `BulkInsertTimeEntriesRequest`
  - Payload shape: `{ "entries": [ { company_id, employee_id, project_id, task_id, date, hours } ] }`
  - `entries` — required, array, min 1 item
  - Per-entry: `company_id` integer exists, `employee_id` integer exists, `project_id` integer exists, `task_id` integer exists, `date` required date, `hours` required numeric > 0
  - `BelongsToCompany` applied to `employee_id`, `project_id`, `task_id` within each entry, passing the sibling `company_id`
  - `EmployeeAssignedToProject` applied using the sibling `employee_id` and `project_id` within each entry

---

### Step 3 — Actions

Create the one Action that contains real business logic; listing is handled inline in controllers.

- [x] `BulkInsertTimeEntriesAction`
  - Wraps everything in a DB transaction
  - Intra-batch check: groups entries by `employee_id` + `date`; fails if any group contains more than one distinct `project_id`
  - DB check: for each entry, queries `time_entries` for the same `employee_id` + `date` with a different `project_id`; fails if found
  - On any failure throws an exception to trigger rollback; returns the created collection on success

---

### Step 4 — Controllers and routes

Wire up thin controllers that delegate entirely to actions and register all routes.

- [x] `CompanyController@index` — filters by `?search=` on name inline, returns `CompanyResource::collection`
- [x] `EmployeeController@index` — route-model-binds `{company}`, returns `EmployeeResource::collection` of its employees
- [x] `ProjectController@index` — route-model-binds `{company}`, returns `ProjectResource::collection` of its projects
- [x] `TaskController@index` — route-model-binds `{company}`, returns `TaskResource::collection` of its tasks
- [x] `TimeEntryController@index` — returns `TimeEntryResource::collection` of all entries
- [x] `TimeEntryController@bulkStore` — injects `BulkInsertTimeEntriesRequest`, calls `BulkInsertTimeEntriesAction`, returns `TimeEntryResource::collection` with `201`
- [x] `routes/api.php` — register all six routes under `/api` prefix

---

### Step 5 — Feature tests

Cover every endpoint and every validation/business-rule path with Pest feature tests.

- [x] `GET /companies` — returns paginated list; `?search=` filters by name
- [x] `GET /companies/{company}/employees` — returns employees scoped to company; 404 on missing company
- [x] `GET /companies/{company}/projects` — same pattern
- [x] `GET /companies/{company}/tasks` — same pattern
- [x] `GET /time-entries` — returns all entries
- [x] `POST /time-entries/bulk` happy path — entries are persisted, 201 with resource collection returned
- [x] Validation failures — missing fields, wrong types
- [x] `BelongsToCompany` failure — employee/project/task from a different company
- [x] `EmployeeAssignedToProject` failure — employee not in project
- [x] DB conflict — employee already has a different project on that date
- [x] Intra-batch conflict — two entries in same payload share employee + date but differ on project
- [x] All-or-nothing — a partial failure leaves zero rows inserted

## Notes

- Cross-field validation within `entries.*` (e.g. `entries.0.employee_id` must belong to `entries.0.company_id`) requires the Rule classes to receive sibling values at construction time; the FormRequest will resolve them in the `rules()` method by iterating over the raw `entries` array.
- `BulkInsertTimeEntriesAction` throws a domain exception on conflict (not a validation exception), which the controller catches and converts to a 422 response with a structured error message.
- Simple listing controllers query the model directly (no action); an action is only justified when there is real business logic to isolate (i.e. `BulkInsertTimeEntriesAction`).
- Route model binding is used for `{company}` on the nested resource routes; a 404 is returned automatically if the company does not exist.
- No authentication is in scope; no middleware beyond the default `api` group is needed.
