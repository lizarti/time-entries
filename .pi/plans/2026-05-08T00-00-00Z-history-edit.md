# History View — editable entries

The History View is currently read-only. This plan adds the full edit flow: a
`PUT /api/time-entries/{id}` endpoint, the action and request class that back it,
a service method and composable update function on the frontend, and an inline
edit mode for each history row (all fields except Company).

## Goal

Every row in the History tab will be editable inline. Clicking Edit turns the row
into a form (date picker + comboboxes for employee/project/task + number input for
hours). Company remains a read-only label. Saving calls the update endpoint and
refreshes the row in place. Cancelling discards all changes.

## Steps

### Step 1 — Backend: request, action, and controller method

Add validation, business-rule check, and the `update()` controller method.

- [x] Create `app/Http/Requests/UpdateTimeEntryRequest.php`
  - Fields: `employee_id`, `project_id`, `task_id`, `date`, `hours` (company_id excluded — not updatable)
  - Reuse `BelongsToCompany` and `EmployeeAssignedToProject` rules; derive company_id from the route model (`$this->route('time_entry')->company_id`)
- [x] Create `app/Actions/TimeEntry/UpdateTimeEntryAction.php`
  - Accept the existing `TimeEntry` model and a validated array
  - Run the same "employee cannot work on a different project on the same date" conflict check, ignoring the current entry's own row
  - Update and return the entry (eager-loaded with company/employee/project/task)
- [x] Add `update()` to `TimeEntryController`
  - Route-model-bind `TimeEntry $timeEntry`
  - Inject `UpdateTimeEntryRequest` and `UpdateTimeEntryAction`
  - Return `TimeEntryResource` with HTTP 200

### Step 4 — Frontend: HistoryEntryRow component and HistoryTab edit mode

Build the editable row component and wire it into the History tab.

- [x] Create `resources/js/components/HistoryEntryRow.vue`
  - Props: `entry: TimeEntry`, `errors: Record<string, string>`
  - Emits: `save(payload: UpdateTimeEntryPayload)`, `cancel()`
  - Internal draft state mirrors the entry's current values (employee_id, project_id, task_id, date, hours)
  - Company column: always a plain text label (locked)
  - Date: same Popover + Calendar picker as `NewTimeEntryRow`
  - Employee / Project / Task: `AppCombobox`, options loaded via `useEmployees` / `useProjects` / `useTasks` keyed to `entry.company.id`
  - Hours: `Input type="number"`
  - Actions column: Save (primary) and Cancel (ghost) buttons
- [x] Update `HistoryTab.vue`
  - Track `editingId: number | null` (which row is currently in edit mode)
  - Track `rowErrors: Record<string, string>` for field-level errors from the API
  - When not editing: render plain text cells + an Edit (pencil icon, ghost) button per row
  - When editing (`entry.id === editingId`): render `<HistoryEntryRow>` instead
  - On `save`: call `update()` from `useTimeEntries`; on success close edit mode; on 422 parse the `errors` object and pass to the row
  - On `cancel`: reset `editingId` and clear errors

## Notes

- Company is intentionally excluded from `UpdateTimeEntryRequest` — changing it would require re-validating every relation and is not in scope.
- The conflict rule for `UpdateTimeEntryAction` must exclude the entry being updated itself (same logic as bulk insert but with an extra `where('id', '!=', $entry->id)` guard).
- Only one row can be in edit mode at a time; opening a second row implicitly cancels the first (or we simply disable Edit on other rows while one is active — whichever feels cleaner, decide during Step 4).
- `HistoryEntryRow` intentionally mirrors `NewTimeEntryRow` in structure; consider extracting shared field components in a follow-up if duplication grows.
- `ApiItem<T>` is needed because Laravel's `JsonResource` wraps a single resource as `{ data: { ... } }`, while `ApiCollection` wraps collections.
